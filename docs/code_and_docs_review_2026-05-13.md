# 代码与文档快速评审建议（2026-05-13）

基于当前仓库的快速走查，项目主线已经从 Stage 1 扩展到 Stage 2.x 规划，测试覆盖了队列、执行器、HTTP 服务和沙箱等关键模块。

## 关键观察

1. `README.md` 里的项目结构仍写着 `.python-version # Python 3.11`，但仓库当前未包含该文件，且 `pyproject.toml` 声明是 `>=3.9`。
2. 根 README 说明了 Stage 2.5 战略文档，但缺少“开发者下一步怎么做”的可执行清单。
3. 当前已有 `scripts/check_doc_consistency.py`，说明文档一致性已经被纳入工程化路径，建议继续强化“单一事实来源”。

## 建议（按优先级）

### P0：统一运行环境描述（避免新同学踩坑）
- 建议明确“推荐开发版本”与“最低兼容版本”：
  - 最低兼容：Python 3.9（与 `pyproject.toml` 一致）
  - 推荐开发：Python 3.11（如团队确实以 3.11 作为日常环境）
- 在 README 明确二者差异，避免用户误以为必须是 3.11。

### P1：补一份“执行路线图”
- 在 `docs/` 增加短文档，聚焦未来 1~2 周的工程动作（不是战略愿景），例如：
  - 先完成哪些 P0 验收项
  - 每项对应哪些测试文件
  - 成功判定命令是什么

### P1：把文档与测试做显式映射
- 建议在 Stage 2/2.5 文档末尾增加“测试映射表”：
  - 目标能力 → 对应测试文件/用例
  - 能让评审快速判断“文档承诺是否被自动化验证”。

### P2：增加本地一键健康检查脚本
- 整合为一个命令（例如 `make check` 或 `uv run ...`），至少包含：
  - 单测
  - 文档一致性检查
  - 基础 lint（可后续补）

## 建议的最小落地顺序

1. 先修 README 的 Python 版本表述（当天可完成）。
2. 新增“短周期执行路线图”文档（半天）。
3. 增加文档-测试映射与一键检查（1~2 天）。
